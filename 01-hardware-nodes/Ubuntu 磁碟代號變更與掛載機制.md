# Ubuntu 磁碟代號變更與掛載機制

## 主文件 (Main Document)

### 摘要

在 Ubuntu 等現代 Linux 作業系統中，新增硬體磁碟後，原有的磁碟代號（如 `/dev/sda` 變更為 `/dev/sdb`）屬於系統動態分配的正常現象。只要系統的 `/etc/fstab` 設定檔採用 **UUID** 或 **LVM ID** 進行掛載，磁碟代號的變更便**不會**對系統運作造成任何負面影響。

### 系統掛載機制解析 (基於 `/etc/fstab` 實例)

傳統的 Linux 系統可能會依賴動態的磁碟代號（`/dev/sdX`）來掛載分區，這在增減硬體時極易導致掛載失敗。現代標準做法是採用通用唯一辨識碼（UUID）或邏輯捲軸（LVM）。

以下為標準安全配置的掛載方式對照表：

| 掛載點 (Mount Point) | 掛載類型 | 掛載識別方式 (Identifier) | 穩定性分析 |
| --- | --- | --- | --- |
| **`/` (根目錄)** | `ext4` (LVM) | `/dev/disk/by-id/dm-uuid-LVM-...` | **高**。鎖定邏輯捲軸的絕對 ID，不受實體磁碟代號影響。 |
| **`/boot`** | `ext4` | `UUID=87a98c52-...` | **高**。精準鎖定分區 UUID。 |
| **`/boot/efi`** | `vfat` | `UUID=6462-8F8A` | **高**。精準鎖定 EFI 系統分區 UUID。 |
| **`/var/lib/rancher`** | `ext4` | `UUID=f536c810-...` | **高**。獨立資料槽位採用 UUID 掛載。 |
| **`swap` (虛擬記憶體)** | `swap` | `/swap.img` (實體檔案) | **高**。不依賴磁碟分割區，直接使用檔案系統內的映像檔。 |

### 硬體層級潛在影響：BIOS/UEFI 開機順序

作業系統層面的掛載不受影響，但**硬體層面**可能會因為偵測到新磁碟，而自動改變預設的開機啟動順位。

* **症狀**：開機後出現黑畫面，或提示 `No bootable device`（找不到可開機裝置）。
* **排除方式**：進入主機板的 BIOS/UEFI 設定介面，將開機優先權（Boot Priority）手動調整回原本包含 Ubuntu 系統的磁碟即可。

## 常見問答 (Q&A)

**Q1：為什麼安裝新磁碟後，原本的 `sda` 會變成 `sdb`？**
**A1：** Linux 核心的裝置命名機制（udev）會根據硬體初始化和匯流排偵測的順序來動態分配代號。新安裝的磁碟如果接在優先級較高的 SATA/NVMe 插槽，系統就會將 `sda` 分配給新磁碟，原有的磁碟則順延為 `sdb`。

**Q2：磁碟代號改變會導致 Ubuntu 無法正常開機或讀取不到資料嗎？**
**A2：** 不會。只要 `/etc/fstab` 檔案中是使用 UUID、LVM ID 或檔案路徑（如 `/swap.img`）來指定掛載點，系統就能精準找到對應的磁碟區塊。作業系統認的是磁碟分區本身的唯一識別碼，而不是 `sda` 或 `sdb` 這種動態代號。

**Q3：我需要修改 `/etc/fstab` 檔案中的註解嗎？（例如：`# /boot was on /dev/sda2...`）**
**A3：** 不需要。這些註解是在系統初始安裝（curtin installation）時由安裝程式自動產生的紀錄，僅供人類閱讀參考，完全不會參與系統的實際運作。

**Q4：如果裝了新硬碟後開不了機，顯示「No bootable device」，該如何處理？**
**A4：** 這是主機板 BIOS/UEFI 的開機順序被新磁碟打亂所致。請重新開機並進入 BIOS/UEFI 設定介面，找到「Boot」或「啟動順序」選單，將原本的系統碟（或名稱為 `ubuntu` 的 UEFI 啟動項目）調整回第一順位，儲存設定後重開機即可。
